# Crypto Rationale

## Обзор модели угроз

Messenger — self-hosted приложение. Сервер является полностью недоверенным: он хранит только зашифрованный ciphertext и не имеет доступа к plaintext сообщениям, ключам сессий или медиафайлам. Компрометация сервера не раскрывает содержимое переписки.

Дополнительные ограничения модели:
- Физический доступ злоумышленника к устройству клиента: vault шифрование защищает локальное хранилище ключей.
- Перехват трафика: TLS 1.3 с Perfect Forward Secrecy исключает расшифровку записанного трафика.
- Атака на провайдер push-уведомлений: VAPID-подписанные уведомления содержат только `chatId`, не plaintext.

## Шифрование сообщений: X3DH + Double Ratchet

### X3DH (Extended Triple Diffie-Hellman)

**Почему:** Forward secrecy для начального handshake; совместимость с Signal Protocol. Позволяет инициатору начать сессию асинхронно, без одновременного присутствия получателя онлайн.

**Как:**
- Identity key (IK), signed prekey (SPK), one-time prekeys (OPK) публикуются на сервере при регистрации устройства.
- Initiator вычисляет общий секрет через четыре DH-операции (IK_i × SPK_r, EK_i × IK_r, EK_i × SPK_r, EK_i × OPK_r).
- Параметры: кривая X25519 через libsodium. Ephemeral key генерируется для каждого нового handshake.
- Session bootstrap — device-scoped: у каждого устройства получателя своя сессия.

### Double Ratchet

**Почему:** Per-message forward secrecy + break-in recovery. Компрометация одного ключа не раскрывает прошлые или будущие сообщения.

**Как:**
- KDF chain ratchet: после каждого отправленного сообщения производный ключ обновляется через HKDF.
- DH ratchet: при получении нового сообщения от собеседника выполняется DH-шаг, что обеспечивает восстановление после компрометации (break-in recovery).
- Skipped message keys кешируются для поддержки out-of-order доставки.

### Sender Keys (групповые чаты)

**Почему:** O(1) шифрование для групп вместо O(n). Каждый участник шифрует сообщение один раз, а не для каждого получателя отдельно.

**Как:**
- Shared chain key хранится у каждого участника группы.
- При добавлении нового участника или выходе члена из группы — sender key rotation.
- Sender Key Distribution Message (SKDM) передаётся новым участникам через пеeper-to-peer E2E канал (X3DH + Double Ratchet).

## Шифрование медиа: AES-256-GCM

**Почему:** Authenticated encryption (шифрование + проверка целостности в одной операции). WebCrypto API нативно поддерживает AES-GCM без внешних зависимостей.

**Параметры:**
- Алгоритм: AES-256-GCM
- Длина ключа: 256 бит (32 байта), генерируется случайно для каждого файла через `crypto.getRandomValues`
- Nonce: 96 бит (12 байт), случайный, prepended к ciphertext
- Auth tag: 128 бит (16 байт)
- Wire format: `nonce (12 байт) || ciphertext || auth_tag`

**Управление ключами:** ключ передаётся получателям в зашифрованном E2E payload сообщения (base64). Сервер хранит только ciphertext и не может расшифровать файл.

Подробная спецификация: [shared/crypto-contracts/aes-gcm-spec.md](../shared/crypto-contracts/aes-gcm-spec.md)

## Vault шифрование: PBKDF2 + AES-256-GCM

**Почему:** Защита локального хранилища (IndexedDB, ключи сессий, device keys) от физического доступа к устройству. Passphrase не покидает клиент.

**Параметры** (из `shared/native-core/crypto/cryptoVault.ts`):
- KDF: PBKDF2, SHA-256, **600 000 итераций**
- Соль: 16 байт, случайная, хранится в `localStorage` под ключом `vault_salt_v1`
- Nonce: 12 байт, случайный, prepended к каждому зашифрованному blob
- Шифрование: AES-256-GCM (ключ 256 бит, derivation через WebCrypto `deriveKey`)

**Компромисс:** PBKDF2 менее устойчив к GPU-атакам, чем Argon2id. Выбран из-за нативной поддержки в WebCrypto API (браузеры не предоставляют Argon2id). WebAuthn PRF запланирован как альтернатива без passphrase.

## Хеширование паролей: Argon2id

**Почему:** Memory-hard алгоритм, устойчив к GPU/ASIC атакам. Рекомендован OWASP для хеширования паролей.

**Параметры** (из `server/internal/password/password.go`):
- Алгоритм: `argon2id`, версия 19
- Memory: **65536 KiB (64 MiB)**
- Iterations (time cost): **3**
- Parallelism: **4**
- Salt: 16 байт, случайная для каждого хеша
- Hash output: 32 байта
- Формат в БД: PHC-string `$argon2id$v=19$m=65536,t=3,p=4$<base64-salt>$<base64-hash>`

**Lazy migration:** bcrypt-хеши (`$2a$`, `$2b$`, `$2y$`) перехешируются в Argon2id при следующем успешном логине пользователя. Функция `NeedsRehash` детектирует устаревший формат.

**Unicode-нормализация:** пароль нормализуется до NFC перед хешированием, чтобы Unicode-пароли давали одинаковый хеш независимо от раскладки клиента.

## Транспортная безопасность: TLS 1.3

**Почему:** Устраняет уязвимости TLS 1.0–1.2 (BEAST, POODLE, CRIME и др.). TLS 1.3 обязателен для Perfect Forward Secrecy на транспортном уровне.

**Конфигурация:** `MinVersion: tls.VersionTLS13` в Go `http.Server`. HTTPS включается при наличии переменных `TLS_CERT`/`TLS_KEY` или при деплое за reverse proxy с `BEHIND_PROXY=true`.

## Push-уведомления: VAPID

**Почему:** Верификация отправителя без учётных данных третьей стороны (без API-ключей push-сервиса). Сервер доказывает своё право отправлять уведомления через криптографическую подпись.

**Как:**
- ECDH P-256 ключевая пара (`VAPID_PUBLIC_KEY` / `VAPID_PRIVATE_KEY`) генерируется при первом запуске и должна быть персистирована.
- Сервер подписывает JWT с VAPID-ключом при каждой отправке push.
- Payload уведомления содержит только `chatId` и тип события — plaintext содержимое сообщений не передаётся.

---

## Известные ограничения и планы

- **WebAuthn PRF** для vault key derivation вместо passphrase — запланировано. Устранит необходимость вводить passphrase вручную при каждом открытии vault.
- **SQLCipher** для нативных клиентов — отложено (см. P2-LOC-2). Нативные клиенты пока не имеют шифрования локальной БД на уровне файла.
- **Perfect Forward Secrecy для медиа:** текущая реализация генерирует случайный ключ для каждого файла — PFS на уровне медиа обеспечивается. ✅
- **VAPID ключи:** push-подписки инвалидируются при перезапуске сервера, если VAPID ключи не персистированы. Обязательно настроить `VAPID_PUBLIC_KEY`/`VAPID_PRIVATE_KEY` перед деплоем в production.
