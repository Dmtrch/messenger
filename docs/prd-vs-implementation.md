# Отчёт: Отличия реализации от PRD

Дата: 2026-04-16
Источники: `prd.md` (требования) ↔ `server/`, `client/`, `shared/`, `apps/` (код).
Markdown-файлы (`docs/**`, `README.md`, `CLAUDE.md`) при сравнении не учитывались — только код.

Легенда: ✅ соответствует · ⚠️ частично · ❌ отсутствует

---

## 1. Резюме

Реализована полнофункциональная PWA-версия мессенджера с E2EE-ядром (X3DH + Double Ratchet через libsodium), WebSocket-хабом, WebRTC 1↔1 звонками и веб-админкой. По существу проект покрывает «ядро» Signal-подобного клиент-серверного обмена, но многие специфические для PRD «Private Node» возможности отсутствуют либо реализованы слабее, чем заявлено.

Главные расхождения:

- инвайты не имеют принудительного TTL = 180 секунд и не выпускаются как QR/ссылка;
- отсутствует серверная дистрибуция нативных бинарей (.exe/.dmg/.apk) — сервер раздаёт только PWA;
- отсутствуют исчезающие сообщения, отложенная отправка, групповые звонки, Local Bot API, Kill Switch / Remote Wipe, retention/квоты, роль «модератор»;
- хеширование паролей выполнено на bcrypt cost=12 вместо требуемого Argon2id;
- локальная БД клиента (idb-keyval) не шифруется (нет SQLCipher);
- multi-device pairing через QR не реализован.

## 3. Система приглашений и регистрации (PRD §3.2)

| Требование PRD | Реализация | Статус |
|---|---|---|
| Только админ генерирует инвайт | `RequireAdmin` middleware на `POST /api/admin/invite-codes` (`server/internal/admin/handler.go:95`) | ✅ |
| Формат: ссылка **и** QR-код | Только текстовый код. UI создаёт инвайт без визуального QR; вход поддерживает `?invite=<code>` URL-параметр (`client/src/pages/AuthPage.tsx:20`) | ⚠️ |
| Жёсткий TTL = 3 минуты (180 с), автоматическая аннуляция | `InviteCode.ExpiresAt int64` опционален; UI вызывает `apiPost('/api/admin/invite-codes', {})` без `expires_at`, сервер при `ExpiresAt == 0` сохраняет код «бессрочным» (`server/db/queries.go:893+`, `server/internal/auth/handler.go:60-64`) | ❌ |
| Кнопка «Аннулировать» | Маршрута удаления/инвалидации инвайта нет | ❌ |
| Регистрация: переход → логин/пароль → редирект на страницу загрузок | Регистрация работает (`/auth/register`), но «страницы загрузок» нет — после успеха пользователь попадает в чаты | ⚠️ |
| Журнал активаций (IP, устройство) | `invite_codes` фиксирует только `used_by`/`used_at`; IP и `User-Agent` не сохраняются | ❌ |

Дополнительно: реализованы режимы регистрации `open|invite|approval` (PRD упоминает только `invite`). Режим `approval` (заявки на регистрацию) — расширение поверх PRD.

## 5. Безопасность и шифрование (PRD §3.5, §1)

### 5.1 E2EE

| Требование | Реализация | Статус |
|---|---|---|
| Double Ratchet | `client/src/crypto/ratchet.ts` + общий `shared/native-core/crypto/`, тесты | ✅ |
| X3DH | `client/src/crypto/x3dh.ts`, `keys` handler на сервере | ✅ |
| Pre-keys в загрузку на сервер | `keys.Handler`, таблицы `identity_keys`, `pre_keys`, `signed_pre_keys` (`POST /api/keys`, `GET /api/keys/{userId}`) | ✅ |
| Ключи хранятся только на устройствах | Сервер хранит публичные ключи; приватные — в браузере (`keystore.ts` через `idb-keyval`) | ✅ |

### 5.2 Транспорт и пароли

| Требование | Реализация | Статус |
|---|---|---|
| TLS 1.3 | TLS включается через `TLS_CERT`/`TLS_KEY` (`server/cmd/server/config.go`); версия не форсируется в коде, но Go ≥1.22 по умолчанию >= TLS 1.2/1.3 | ⚠️ |
| **Argon2id** для паролей | Используется **bcrypt cost=12** (`server/internal/auth/handler.go:80,213,323`) | ❌ |
| AES-256-GCM (симметричное шифрование) | Шифрование сообщений в `libsodium` (XChaCha20-Poly1305 `secretbox`/`box`); AES-GCM не используется | ⚠️ |

### 5.3 Локальное хранилище клиента (PRD §1.2.2 и отдельный блок «Локальное хранилище»)

| Требование | Реализация | Статус |
|---|---|---|
| SQLCipher AES-256 для локальной БД | PWA использует **idb-keyval / IndexedDB** без шифрования (`client/src/store/messageDb.ts:9`) | ❌ |
| Hardware-Backed ключ (Keychain/Keystore/Secure Enclave) | В PWA фундаментально невозможно через стандартный API; нативные клиенты — `apps/mobile/ios/.../crypto/KeyStorage.swift` использует Keychain, Android/Desktop — простое файловое/SQLDelight хранилище без Keystore-обёртки | ⚠️ |
| Биометрия (FaceID/TouchID) для разблокировки | Не реализовано ни в PWA, ни в нативных клиентах | ❌ |
| Encrypted Blobs медиа на устройстве с per-file ключом | Медиа хранятся на сервере и скачиваются по URL; локальное шифрование blob отсутствует | ❌ |
| Zeroing Out при удалении | `idb-keyval` `del/set` без перезаписи; SQLite на сервере — обычные `DELETE` | ❌ |

### 5.4 Голос/видео

| Требование | Реализация | Статус |
|---|---|---|
| DTLS/SRTP | Используется через WebRTC API браузера/нативной реализации — DTLS-SRTP включён в стек по умолчанию | ✅ |
| P2P + TURN fallback | `iceServersHandler` отдаёт STUN + опциональный TURN с временными HMAC-credentials (`server/cmd/server/calls.go`) | ✅ |

## 9. Сценарий привязки нового устройства (PRD §«Multi-device sync»)

PRD описывает сценарий: «Ожидание авторизации сессии» → QR на десктопе → сканирование смартфоном → передача `Ratchet keys` через E2E tunnel → опциональная история.

В коде:

- `keys.Handler` поддерживает регистрацию ключей нового устройства (`POST /api/keys` с `deviceId`);
- но отсутствуют: state «awaiting device approval», временные QR-сессии, передача `Ratchet keys` device-to-device, Re-keying при удалении устройства, UI «Привязать новое устройство», endpoint управления списком устройств в Settings.

Статус: ❌ сценарий из PRD не реализован.

## 11. Что есть в коде сверх PRD

- Режим регистрации `approval` с воркфлоу заявок (`request-register`, `registration-requests`, `approve/reject`).
- Запрос на сброс пароля админом (`password-reset-requests`).
- Native push для iOS (APNs `.p8`) и Android (FCM legacy) на сервере.
- Rate-limiter на auth endpoints (20 req/min/IP).
- WebSocket-протокол с поддержкой `senderDeviceId` и адресной доставкой `DeliverToDevice`.
- Сборка нативных клиентов (`apps/desktop` Compose Multiplatform, `apps/mobile/android` Kotlin/Compose, `apps/mobile/ios` SwiftUI) — фундамент, не продукт; код звонков и сигналинга уже есть.
- `shared/native-core` — общий TS-рантайм для нативных клиентов с собственными тестами.

## 12. Сводная карта ошибок относительно PRD

| Категория | Расхождения |
|---|---|
| **Критические** | Нет TTL=180с для инвайтов; нет QR-инвайтов; bcrypt вместо Argon2id; нет SQLCipher на клиенте; нет дистрибуции нативных бинарей с сервера; нет Kill Switch / Remote Wipe |
| **Высокие** | Нет multi-device QR-pairing; нет групповых звонков; нет исчезающих сообщений; нет квот/retention/мониторинга; нет роли «модератор» и suspend/ban; нет Local Bot API |
| **Средние** | Нет авто-обновления клиентов; нет лимита размера файла в конфиге; нет встроенной галереи; нет специализированных voice-notes; нет zero-config вшитого адреса сервера |
| **Низкие** | Нет журнала активаций по инвайтам с IP/устройством; нет принудительного TLS 1.3 в коде; нет zeroing out при удалении; нет ограничения создания групп админом |

## 13. Рекомендации по приоритетам

1. Приоритет 1 (заявленные ключевые свойства Private Node):
   - Принудительный TTL=180с по умолчанию для инвайтов + аннуляция; QR-генератор в админке.
   - Замена bcrypt → Argon2id (с миграцией хешей).
   - Дистрибуция нативных бинарей через защищённую зону + версионирование.
   - Kill Switch / suspend / revoke-all-sessions.
2. Приоритет 2 (UX-параметры приватности):
   - Исчезающие сообщения с фоновой задачей очистки.
   - Multi-device pairing через QR + передача Ratchet state.
   - Шифрование локальной БД клиента (минимум — passphrase-derived ключ для IndexedDB на PWA, SQLCipher для нативных клиентов).
3. Приоритет 3 (масштабирование):
   - Групповые звонки (SFU или mesh).
   - Local Bot API.
   - Квоты/retention/мониторинг ресурсов в админке.
