# Нереализованные задачи по `docs/superpowers/specs/messenger-spec.md`

## Назначение файла

Этот файл фиксирует требования из [`docs/superpowers/specs/messenger-spec.md`](/Users/dim/vscodeproject/messenger/docs/superpowers/specs/messenger-spec.md), которые на текущий момент не реализованы полностью или реализованы частично.

Формат:

- `Приоритет`
- `Нереализованная задача`
- `Методика реализации`

Шкала приоритетов:

- `Must` — обязательно для соответствия MVP-спецификации или критично для безопасности
- `Should` — желательно для завершённости архитектуры и эксплуатационной надёжности
- `Could` — полезно, но может быть отложено после закрытия критичных разрывов

## Must

### 1. Полноценная модель нескольких устройств

**Приоритет**

`Must`

**Статус**: ⚠️ Фундамент заложен (этапы 3, 7). Таблица `devices` создана; `identity_keys` получила composite PK `(user_id, device_id)` через migration #7; `PopPreKey` фильтрует по `device_id`; `POST /api/keys/register` идемпотентен по IK public key. Остаётся: `GET /api/keys/:userId` возвращает только первое устройство; per-device ratchet state на клиенте не реализован; WS Hub не отслеживает соединения на уровне device_id.

**Нереализованная задача**

Спецификация требует регистрацию нескольких устройств одного пользователя с публикацией identity keys, signed prekeys и one-time prekeys на уровне устройства.

**Методика реализации**

1. Ввести сущность `devices` в БД:
   - `id`
   - `user_id`
   - `device_name`
   - `created_at`
   - `last_seen_at`
2. Перенести `identity_keys` и `one_time_prekeys` с привязки к `user_id` на привязку к `device_id`.
3. Изменить API регистрации и логина так, чтобы клиент создавал или выбирал устройство.
4. Изменить `GET /api/keys/:userId`, чтобы он возвращал набор bundle по всем активным устройствам пользователя.
5. На клиенте хранить отдельные ratchet/session state для пары `chatId + peerUserId + peerDeviceId`.
6. В WebSocket Hub отслеживать соединения на уровне устройства, а не только пользователя.

### 2. Endpoint `POST /api/keys/register`

**Приоритет**

`Must`

**Статус**: ✅ Реализовано (этапы 3, 7). Endpoint существует под JWT, принимает `deviceName`, `ikPublic`, `spkPublic`, `spkSignature`, `opkPublics`. Создаёт `devices` запись, сохраняет `identity_keys` с device_id, сохраняет OPK с привязкой к устройству. Идемпотентен: повторный вызов с тем же `ikPublic` переиспользует существующий `device_id` вместо создания нового устройства.

**Нереализованная задача** *(была)*

В спецификации есть отдельный endpoint `POST /api/keys/register` для регистрации набора публичных ключей. В текущей реализации сохранение ключей происходило только внутри `POST /api/auth/register`.

**Методика реализации**

1. Добавить отдельный handler `POST /api/keys/register` под JWT.
2. Разрешить повторную публикацию:
   - нового signed prekey;
   - новой пачки one-time prekeys;
   - ключей для нового устройства.
3. Разделить сценарии:
   - первичная регистрация пользователя;
   - повторная регистрация ключей устройства;
   - ротация ключей.
4. На клиенте вызывать этот endpoint:
   - после логина на новом устройстве;
   - при low-prekey состоянии;
   - при ротации signed prekey.

### 3. Полноценные Sender Keys для групп

**Приоритет**

`Must`

**Статус**: ✅ Реализовано в этапе 5 (ветка `feature/stage5-crypto`, коммит `b5d76bd`). Lazy SKDM при первом сообщении, симметричный рэтчет, `senderkey.ts`, IndexedDB-хранение.

**Нереализованная задача**

Спецификация требует Sender Keys для групповых чатов. Сейчас клиент шифрует одно и то же сообщение отдельно для каждого получателя.

**Методика реализации**

1. Ввести group sender key state на клиенте:
   - `group_id`
   - `sender_key_id`
   - `chain_key`
   - подпись или верификация распределения ключа.
2. Добавить типы системных group-control сообщений:
   - раздача Sender Key;
   - ротация Sender Key;
   - присоединение нового участника;
   - исключение участника.
3. Сохранять один ciphertext сообщения на чат, а не отдельную копию на каждого получателя.
4. Для нового участника или после изменения состава выполнять пересоздание Sender Key.
5. Обновить криптографический слой клиента:
   - отдельный модуль `senderKeys.ts`;
   - восстановление состояния после reload.

### 4. Полная реализация Double Ratchet со skipped message keys

**Приоритет**

`Must`

**Статус**: ✅ Реализовано в этапе 5. `RatchetState` расширен полями `prevSendCount` и `skippedKeys`, кэш ограничен MAX_SKIP=100, скипованные ключи сериализуются в IndexedDB.

**Нереализованная задача**

В `client/README.md` и по факту кода отсутствует полноценная поддержка skipped message keys, необходимых для корректной работы при сообщениях вне порядка доставки.

**Методика реализации**

1. Расширить `RatchetState`:
   - хранение `skippedKeys`;
   - лимит объёма кэша;
   - TTL для старых ключей.
2. При decrypt:
   - сначала искать подходящий skipped key;
   - только потом выполнять продвижение chain.
3. Сериализовать skipped keys в IndexedDB.
4. Добавить тесты:
   - сообщения пришли в неправильном порядке;
   - дубликаты сообщений;
   - пропущенные сообщения.

### 5. Encrypted media at rest

**Приоритет**

`Must`

**Статус**: ✅ Реализовано в этапе 2 (JWT, mediaId, media_objects) и этапе 5 (client-side XSalsa20-Poly1305, ключ в payload, расшифровка на скачивании).

**Нереализованная задача**

Спецификация требует, чтобы медиафайлы загружались на сервер уже в зашифрованном виде и оставались зашифрованными на диске. Сейчас сервер хранит обычные файлы.

**Методика реализации**

1. На клиенте шифровать файл перед upload:
   - случайный media key;
   - nonce или iv;
   - ciphertext blob.
2. Передавать на сервер:
   - зашифрованный бинарный файл;
   - `mimeType`;
   - `size`;
   - метаданные для последующего получения.
3. На сервере хранить только ciphertext и метаданные.
4. Ссылку на `mediaId`, ключ шифрования и MIME включать внутрь зашифрованного message payload.
5. При скачивании клиент должен:
   - получить ciphertext;
   - расшифровать локально;
   - создать `blob:` URL.

### 9. Оффлайн-режим просмотра истории

`Must`

**Методика реализации**

   - IndexedDB таблицы для чатов;

   - состояния синхронизации.
2. При получении и отправке сообщений сохранять их не только в Zustand, но и в IndexedDB.
3. При старте приложения сначала читать локальный кэш, затем делать background sync.
4. Добавить признак stale или offline версии истории.

## Could (звонки)

### 28. Аудио/видео звонки 1-на-1

**Приоритет**

`Could`

**Нереализованная задача**

Текущая реализация не содержит никаких компонентов для аудио/видео связи. Нет WebRTC-сигнализации, нет UI звонков, нет STUN/TURN конфигурации.

**Методика реализации**

1. Добавить WS-события сигнализации на сервере:
   - `call_offer`, `call_answer`, `ice_candidate` — транзитная пересылка между участниками
   - `call_end`, `call_reject`, `call_busy` — управление состоянием звонка
   - Сервер не обрабатывает медиа — только пересылает сигналы
2. Хранить состояние активных звонков в Hub (in-memory map `callId → participants`)
3. На клиенте реализовать:
   - `RTCPeerConnection` с публичным STUN-сервером
   - UI входящего звонка (ringtone, принять/отклонить)
   - UI активного звонка (камера, микрофон, завершить)
   - Обработку ICE negotiation и reconnect
4. Добавить конфигурацию STUN/TURN через env `STUN_URL`, `TURN_URL`, `TURN_CREDENTIAL`

### 29. Поддержка TURN-сервера

**Приоритет**

`Could`

**Нереализованная задача**

Без TURN ~15–30% звонков не установятся (симметричный NAT, корпоративный firewall). Публичные TURN-серверы нестабильны и небезопасны для приватного мессенджера.

**Методика реализации**

1. Поднять coturn на том же хосте или отдельном VPS
2. Передавать TURN credentials клиенту через отдельный endpoint `GET /api/calls/ice-servers` (JWT required)
3. Credentials должны быть временными (TURN secret + username = timestamp, TTL 24ч)
4. Документировать требования к upload-bandwidth: видео 720p ≈ 1–2 Мбит/с на звонок через TURN

### 30. Групповые аудио/видео звонки (SFU)

**Приоритет**

`Could`

**Нереализованная задача**

WebRTC mesh (каждый ↔ каждый) нагружает клиент квадратично: при 5 участниках каждый шлёт и принимает 4 потока. Для групп нужен SFU.

**Методика реализации**

1. Выбрать SFU: LiveKit (Go, self-hosted, Docker) — оптимально для данного стека
2. Интегрировать LiveKit SDK на клиенте вместо прямого `RTCPeerConnection`
3. Сервер выдаёт JWT-токены доступа к LiveKit room через `POST /api/calls/room`
4. LiveKit разворачивается как отдельный Docker-сервис
5. Ограничение: требует отдельного сервиса и значительных ресурсов (CPU для транскодинга опционально)
