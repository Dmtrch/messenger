# iOS Update Policy для Self-Hosted Messenger

## Почему iOS не поддерживает прямую установку IPA

На iOS система безопасности Apple запрещает установку приложений из произвольных источников. В отличие от Android (APK sideloading), пользователь не может скачать `.ipa`-файл по ссылке и установить его — iOS принимает только подписанные пакеты через один из официальных каналов Apple.

**Технические причины:**
- Все бинарники должны быть подписаны Apple-сертификатом (code signing), который проверяется при установке.
- `UIApplication` не имеет API для программной установки приложений (в отличие от Android `PackageInstaller`).
- Sandbox и entitlements блокируют любые операции записи вне песочницы приложения.
- `itms-services://` (старый OTA для Enterprise) требует подписанного профиля и Enterprise-сертификата Apple Developer Program (>$299/год).

Итог: метод `downloadAndInstall`, реализуемый на Android через `PackageInstaller`, **невозможен на iOS**.

## Три варианта дистрибуции для Self-Hosted мессенджера

### 1. TestFlight

**Описание:** Официальная бета-платформа Apple. Приложение загружается в App Store Connect, пользователи получают приглашение по email или публичной ссылке.

| | |
|---|---|
| **Плюсы** | Бесплатно в рамках Apple Developer Program ($99/год). До 10 000 внешних тестировщиков. Обновления пуш-уведомлением в приложении TestFlight. Apple проводит упрощённый review (быстрее, чем App Store). |
| **Минусы** | Требует Apple Developer Program. Каждая сборка проходит review Apple (1–3 дня). Сборки действительны 90 дней — нужно регулярно выпускать новые. Пользователь устанавливает TestFlight отдельно. Не подходит для полностью закрытого корпоративного деплоя. |

**Вывод для self-hosted:** Лучший вариант для небольших команд (< 10 000 человек). Обновление происходит через уведомление TestFlight — пользователь нажимает «Обновить».

---

### 2. App Store

**Описание:** Публичный магазин Apple. Приложение доступно всем пользователям iOS после прохождения review.

| | |
|---|---|
| **Плюсы** | Максимальный охват. Автоматические обновления (если включены у пользователя). Встроенная система обновлений. |
| **Минусы** | Строгий review Apple (privacy, функциональность, брендинг). Self-hosted сервер должен соответствовать App Store Guidelines — нельзя скрывать функциональность. Нельзя публиковать приложение только для внутреннего использования без соответствия публичным гайдлайнам. Публично видно всем. |

**Вывод для self-hosted:** Подходит, если мессенджер планируется как публичный продукт. Для закрытого корпоративного использования — нежелательно (риск отклонения review из-за ограниченной аудитории).

---

### 3. Enterprise MDM (Apple Business Manager / Mobile Device Management)

**Описание:** Apple Developer Enterprise Program ($299/год) позволяет распространять приложения внутри организации через MDM-систему (Jamf, Microsoft Intune, VMware Workspace ONE и др.) без App Store.

| | |
|---|---|
| **Плюсы** | Полный контроль над дистрибуцией — не нужен App Store. Обновления через MDM без участия пользователя. Приложение видно только сотрудникам. Подходит для корпоративного self-hosted деплоя. |
| **Минусы** | Требует Apple Developer Enterprise Program ($299/год). Требует MDM-инфраструктуры (Jamf и т.д.) — дополнительные затраты. Apple регулярно проверяет соответствие программе — злоупотребления (дистрибуция вне организации) ведут к отзыву сертификата. Установка на устройства пользователей требует их регистрации в MDM. |

**Вывод для self-hosted:** Идеальный вариант для организаций с ≥50 сотрудников и существующей MDM-инфраструктурой. Для частных/небольших деплоев — избыточно дорого.

## Стратегия обновлений: Deep-link на App Store как fallback

Поскольку `downloadAndInstall` невозможен на iOS, при `hasUpdate == true` приложение показывает баннер, который направляет пользователя в App Store или TestFlight.

### Схема работы

```
GET /api/version
    ↓
{ latestVersion: "1.2.0", forced: false, appStoreUrl: "https://apps.apple.com/app/idXXXXXXXXX" }
    ↓
latestVersion > currentVersion?
    ↓ да
Показать UpdateBannerView
    ↓ нажатие "Обновить"
UIApplication.shared.open(appStoreUrl)
```

### URL-схемы для открытия App Store

```swift
// Предпочтительный deep-link (открывает App Store напрямую)
itms-apps://itunes.apple.com/app/idXXXXXXXXX

// Fallback через браузер (работает везде, включая симулятор)
https://apps.apple.com/app/idXXXXXXXXX
```

Алгоритм выбора URL:
1. Попробовать открыть `itms-apps://...` через `UIApplication.canOpenURL`.
2. Если недоступно (симулятор, ограничения MDM) — открыть `https://apps.apple.com/...`.

### Поведение при `forced == true`

Если сервер вернул `forced: true` (критическое обновление безопасности):
- Баннер блокирует весь UI (sheet/fullScreenCover, без кнопки «Закрыть»).
- Пользователь не может пользоваться приложением до обновления.
- Применяется та же deep-link стратегия.

## Почему нельзя делать `downloadAndInstall` как на Android

| Механизм | Android | iOS |
|---|---|---|
| Установка из файла | `PackageInstaller` API | Недоступно |
| Подпись пакета | Ключ разработчика (любой) | Только сертификат Apple |
| OTA без store | Да (APK sideloading) | Только Enterprise MDM |
| API для обновления | `ACTION_INSTALL_PACKAGE` Intent | Нет аналога |

Apple намеренно не предоставляет публичный API для программной установки приложений — это ключевое ограничение безопасности iOS. Единственный обходной путь — направить пользователя в официальный канал (App Store, TestFlight, MDM), где Apple контролирует процесс установки.
