# Установка сервера Messenger — пошаговое руководство

Это руководство для обычного пользователя. Оно объясняет **с нуля**, как поднять
собственный сервер Messenger. Технических знаний почти не требуется — достаточно
уметь копировать команды в терминал.

> **Что такое сервер Messenger?**
> Это программа, которую вы запускаете на своём компьютере или на арендованном
> сервере (VPS). Все сообщения, звонки и файлы идут только через него — никаких
> чужих облаков. Веб-версия мессенджера (PWA) уже встроена внутрь сервера, её не
> нужно ставить отдельно.

---

## 1. Что вам понадобится

| Что | Зачем | Где взять |
|---|---|---|
| **Компьютер или VPS** | На нём будет работать сервер. Подойдёт любой Linux, macOS, Windows или VPS за ~3–5 $/мес. | Например Hetzner, Timeweb, Selectel, любой хостинг |
| **Docker** | Самый простой способ запуска (рекомендуется) | https://www.docker.com/products/docker-desktop/ (или `apt install docker.io` на Linux) |
| **Домен (необязательно)** | Чтобы заходить по красивому адресу `chat.example.com` и иметь HTTPS | Любой регистратор доменов |

Если вы хотите, чтобы мессенджером могли пользоваться люди **через интернет**, а не
только в локальной сети — понадобится домен или бесплатный туннель (см. раздел 6).

### Получите проект на сервер

Способы A, B и C собирают сервер из исходников, поэтому сначала склонируйте
репозиторий на машину-сервер:

```bash
git clone https://github.com/Dmtrch/messenger.git
cd messenger
```

> Если на сервере нет `git`, установите его (`apt install git` на Debian/Ubuntu,
> `dnf install git` на Fedora/RHEL) либо скачайте архив проекта с GitHub
> (кнопка **Code → Download ZIP**) и распакуйте.
>
> Не хотите переносить весь репозиторий? Серверу достаточно одного готового
> бинаря — см. **Способ D** (раздел 5).

---

## 2. Способ A — автоматическая установка (проще всего)

В репозитории есть готовый скрипт, который сам всё настроит и спросит только самое
необходимое (логин/пароль администратора и режим регистрации).

**Шаг 1.** Откройте терминал в папке с проектом Messenger (`cd messenger` после клонирования).

**Шаг 2.** Запустите:

```bash
# macOS / Linux
chmod +x install-server.sh
./install-server.sh
```
```bat
:: Windows
install-server.bat
```

**Шаг 3.** Скрипт спросит:
- режим регистрации (`open` / `invite` / `request` — см. раздел 7);
- логин и пароль администратора;
- (необязательно) TURN-сервер для звонков за сложным NAT.

**Шаг 4.** Готово. Скрипт сам:
- сгенерирует секретный ключ `JWT_SECRET`;
- создаст файл `.env`;
- запустит сервер в Docker;
- сохранит все данные администратора в файл **`server-main.txt`** — сохраните его в надёжном месте.

После этого откройте в браузере `http://localhost:8080` (или адрес вашего сервера) —
вы увидите мессенджер. Админ-панель доступна по адресу `/admin/`.

---

## 3. Способ B — вручную через Docker Compose

Если хотите контролировать каждый шаг.

**Шаг 1. Создайте файл `.env`.** Скопируйте пример и откройте его в редакторе:

```bash
cp .env.example .env
```

**Шаг 2. Задайте обязательный секрет.** Сгенерируйте случайный ключ и впишите его
в `.env` в строку `JWT_SECRET=`:

```bash
openssl rand -hex 32
```

Минимально в `.env` нужно заполнить:

```ini
JWT_SECRET=<вставьте сюда сгенерированный ключ>
REGISTRATION_MODE=invite        # open / invite / request
ADMIN_USERNAME=admin
ADMIN_PASSWORD=<надёжный пароль минимум 8 символов>
```

**Шаг 3. Запустите сервер:**

```bash
docker compose up -d
```

Первая сборка займёт пару минут (Docker сам соберёт клиент и сервер).

**Шаг 4. Проверьте, что работает:**

```bash
docker compose ps          # статус контейнера
docker compose logs -f     # логи (Ctrl+C для выхода)
```

Откройте `http://localhost:8080`.

> **Важно про push-уведомления.** При первом запуске сервер генерирует ключи VAPID
> и печатает их в логах. Скопируйте `VAPID_PUBLIC_KEY` и `VAPID_PRIVATE_KEY` из логов
> в `.env`, иначе после перезапуска push-подписки перестанут работать.

**Данные** (база и медиафайлы) хранятся в Docker-томе `messenger_data` (`/data`
внутри контейнера) и не теряются при перезапуске.

---

## 4. Способ C — без Docker (нативный бинарь)

Если Docker недоступен, можно запускать собранный бинарь напрямую. В репозитории
есть скрипт управления `server-ctl.sh`:

```bash
chmod +x server-ctl.sh

./server-ctl.sh build      # собрать сервер (нужен Go 1.23+)
./server-ctl.sh start      # запустить
./server-ctl.sh status     # статус и порт
./server-ctl.sh logs       # смотреть логи
./server-ctl.sh restart    # перезапустить
./server-ctl.sh stop       # остановить
```

Файл `.env` нужен такой же, как в Способе B. На Windows используйте `server-ctl.bat`.

## 5. Способ D — без репозитория (готовый бинарь или образ)

Исходный код на сервере **не обязателен**. Веб-клиент (PWA) вшит прямо внутрь
серверного бинаря, поэтому серверу для работы достаточно одного файла.

### Вариант 1. Один готовый бинарь

Подходит, если у вас есть собранный бинарь под нужную ОС (например
`messenger-linux-amd64`, собирается локально или через CI — см. раздел 6 в
`Установка-приложений.md`).

**Шаг 1.** Скопируйте на сервер всего два файла:
- бинарь сервера (например `messenger-linux-amd64`);
- файл `.env` (как минимум с `JWT_SECRET`).

**Шаг 2.** Сделайте бинарь исполняемым и запустите:

```bash
chmod +x messenger-linux-amd64
# переменные можно задать в .env или прямо в команде:
JWT_SECRET=$(openssl rand -hex 32) PORT=8080 \
  DB_PATH=./messenger.db MEDIA_DIR=./media \
  ./messenger-linux-amd64
```

База данных и папка медиа создадутся рядом автоматически. Для автозапуска
оформите бинарь как сервис (systemd на Linux) — пример unit-файла есть в
`docs/deployment.md`.

> Выберите бинарь под архитектуру сервера: `messenger-linux-amd64` (обычный VPS),
> `messenger-linux-arm64` (ARM-сервер, Raspberry Pi), `messenger-windows-amd64.exe`,
> `messenger-darwin-arm64` (Mac).

### Вариант 2. Готовый Docker-образ из реестра

Соберите образ один раз (на CI или другой машине) и опубликуйте в реестре
(Docker Hub, GHCR и т. п.). На сервере исходники не нужны — только `docker`:

```bash
# на сервере
docker pull <реестр>/messenger:1.0.0
docker run -d --name messenger \
  -p 8080:8080 \
  -e JWT_SECRET=<секрет> \
  -v messenger_data:/data \
  <реестр>/messenger:1.0.0
```

Данные хранятся в томе `messenger_data` (`/data` внутри контейнера).

> Готовые бинари и образы не лежат в git-репозитории — их создают через CI
> (`build-native.yml` / сборка образа) либо передают вручную.

## 6. Доступ через интернет (домен и HTTPS)

Чтобы мессенджером пользовались с телефонов и из других сетей, серверу нужен
публичный адрес и HTTPS. Два простых варианта:

### Вариант 1. Cloudflare Tunnel (без белого IP и без настройки TLS)

Подходит, если у вас нет внешнего IP (домашний компьютер, сервер за NAT).

1. Создайте туннель: https://one.dash.cloudflare.com → Zero Trust → Networks → Tunnels.
2. Скопируйте **токен туннеля** и впишите в `.env`: `TUNNEL_TOKEN=...`.
3. Поставьте `BEHIND_PROXY=true` в `.env`.
4. Запустите вместе с туннелем:

```bash
docker compose --profile cloudflare up -d
```

Cloudflare выдаст HTTPS-адрес автоматически.

### Вариант 2. Свой домен + прямой TLS

1. Направьте A-запись домена на IP вашего сервера.
2. Получите сертификат (например, через Let's Encrypt / certbot).
3. В `.env` укажите:

```ini
ALLOWED_ORIGIN=https://chat.example.com
TLS_CERT=/path/to/fullchain.pem
TLS_KEY=/path/to/privkey.pem
```

(Пути к сертификатам монтируйте в контейнер через `volumes:` в `docker-compose.yml`.)

> Если перед сервером стоит nginx/Caddy/Cloudflare — поставьте `BEHIND_PROXY=true`,
> а TLS терминируйте на прокси.

## 7. Первый вход и режимы регистрации

Откройте админ-панель: `https://ваш-адрес/admin/` и войдите под логином/паролем
администратора из `.env` (или из `server-main.txt`).

**Режим регистрации** (`REGISTRATION_MODE` в `.env`) определяет, кто может заводить аккаунты:

| Режим | Кто может зарегистрироваться |
|---|---|
| `open` | Кто угодно по ссылке на сервер |
| `invite` | Только по одноразовому инвайт-коду (TTL 3 минуты, генерируется в админке, есть QR) |
| `request` | По заявке, которую одобряет администратор |

Для приватного сервера рекомендуется `invite` или `request`.

## 8. Управление сервером

**Docker:**
```bash
docker compose restart      # перезапуск
docker compose down         # остановить
docker compose logs -f      # логи
docker compose pull && docker compose up -d --build   # обновить
```

**Резервная копия** (важно!). Достаточно сохранить Docker-том с данными:
```bash
docker run --rm -v messenger_messenger_data:/data -v "$PWD":/backup alpine \
  tar czf /backup/messenger-backup.tar.gz -C /data .
```
Подробные процедуры бэкапа/восстановления и ротации ключей — в `docs/release-checklist.md`.

## 9. Все переменные окружения

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `JWT_SECRET` | **обязательно** | Секрет подписи токенов (≥32 символа) |
| `PORT` | `8080` | Порт HTTP |
| `DB_PATH` | `./messenger.db` (в Docker `/data/messenger.db`) | Путь к базе SQLite |
| `MEDIA_DIR` | `./media` (в Docker `/data/media`) | Папка для файлов |
| `REGISTRATION_MODE` | `open` | `open` / `invite` / `request` |
| `ADMIN_USERNAME` / `ADMIN_PASSWORD` | пусто | Создаёт админа при первом запуске |
| `ALLOWED_ORIGIN` | пусто | Разрешённый origin (CORS + WebSocket) |
| `BEHIND_PROXY` | `false` | `true`, если за reverse proxy/туннелем |
| `WS_CONNECT_RATE_LIMIT` | `30` | Лимит WebSocket-подключений с одного IP в минуту |
| `TLS_CERT` / `TLS_KEY` | пусто | Прямой TLS (если без прокси) |
| `VAPID_PUBLIC_KEY` / `VAPID_PRIVATE_KEY` | автогенерация | Ключи Web Push (сохраните из логов!) |
| `TUNNEL_TOKEN` | пусто | Токен Cloudflare Tunnel (профиль `cloudflare`) |
| `STUN_URL` | `stun:stun.l.google.com:19302` | STUN для звонков |
| `TURN_URL` / `TURN_SECRET` | пусто | TURN для звонков за симметричным NAT |

> Полный пример со всеми комментариями — в файле `.env.example`.

## 10. Если что-то не работает

| Симптом | Что проверить |
|---|---|
| `JWT_SECRET is required` | Не заполнен `JWT_SECRET` в `.env` |
| Сервер не открывается в браузере | `docker compose ps` — запущен ли контейнер; правильный ли порт |
| Push-уведомления отвалились после перезапуска | Не сохранены VAPID-ключи из логов в `.env` |
| Не работают звонки между разными сетями | Нужен TURN-сервер (`TURN_URL` / `TURN_SECRET`) |
| HTTPS/домен не отвечает | Проверьте DNS, `ALLOWED_ORIGIN`, `BEHIND_PROXY` |

Логи — главный источник информации: `docker compose logs -f` или `./server-ctl.sh logs`.
