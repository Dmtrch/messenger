# План устранения замечаний по итогам аудита PRD

Дата составления: 2026-04-22
Статус документа: рабочий план
Основание: аудит соответствия `prd.md`

## 1. Цель

Этот документ фиксирует подробный план устранения замечаний, выявленных при сверке кода проекта с `prd.md`, с декомпозицией до подзадач, указанием затрагиваемых файлов, ожидаемого результата и чеклистом контроля исполнения.

Документ предназначен для:

- планирования работ по доведению проекта до требований PRD;
- пошагового контроля выполнения;
- фиксации даты и времени фактического устранения каждого замечания;
- снижения риска пропуска скрытых зависимостей между задачами.

## 2. Правила ведения этого плана

- В колонках с датой и временем указывается фактический момент завершения замечания.
- Если задача выполнена частично, в статусе ставится `partial`, а в примечании фиксируется остаток.
- Если реализация ушла в другой файл или модуль, это дописывается в строку подзадачи.
- После закрытия каждого крупного блока нужно обновить итоговый аудит соответствия PRD.

## 3. Приоритеты

### P0. Критично для PRD и безопасности

- Жесткий TTL инвайтов 180 секунд.
- Исправление логики инвайта на уровне потока активации.
- Полная реализация `remote wipe` на всех клиентах.
- Устранение зависимости по умолчанию от внешних сервисов, противоречащих PRD.
- Проверка членства в чате для REST API групповых звонков.

### P1. Существенно для соответствия PRD

- Реализация key backup / key sync для нового устройства.
- Усиление zero-config.
- Сквозная индикация частного сервера в UX.
- Реализация отложенной отправки сообщений.

### P2. Архитектурное усиление и доведение security-модели

- Защита локальных БД на Android / iOS / Desktop.
- Приведение desktop key storage к более защищенной модели.
- Дополнительные тесты и контрольные сценарии.

## R1. Жесткий TTL инвайта 180 секунд

### Цель

Привести систему инвайтов к строгому TTL `180 секунд` без настраиваемого диапазона, если следовать PRD буквально.

### Затрагиваемые файлы

- `server/internal/admin/handler.go`
- `server/internal/admin/invite_test.go`
- `server/internal/integration/invite_flow_test.go`
- `client/src/pages/AdminPage.tsx`

### Подзадачи

1. Убрать диапазон `60..600` и оставить только фиксированный TTL `180`.
   Задача:
   Изменить серверную логику создания инвайта так, чтобы `ttlSeconds` не принимался из запроса или игнорировался полностью.

2. Упростить контракт API создания инвайта.
   Задача:
   Сделать серверный ответ и документацию коду консистентными: TTL всегда `180`.

3. Обновить админский UI.
   Задача:
   Убедиться, что UI не предполагает кастомный TTL и показывает фиксированный таймер.

4. Обновить тесты.
   Задача:
   Переписать тесты границ TTL на тесты фиксированного значения и поведения истечения.

### Критерий готовности

- Любой новый инвайт живет ровно `180` секунд.
- Клиент и сервер не имеют расхождений по TTL.
- Старые тесты на диапазон удалены или переписаны.

## R2. Приведение lifecycle инвайта к требованиям PRD

### Цель

Сделать поведение инвайта соответствующим сценарию PRD: токен существует ограниченное время и корректно проходит этапы создания, перехода по ссылке, регистрации, аннулирования и истечения.

### Затрагиваемые файлы

- `server/internal/auth/handler.go`
- `server/internal/admin/handler.go`
- `server/db/schema.go`
- `server/db/queries.go`
- `client/src/pages/AuthPage.tsx`
- `server/internal/integration/invite_flow_test.go`

### Подзадачи

1. Формализовать lifecycle инвайта.
   Задача:
   Решить, нужен ли отдельный статус "activated/opened" до регистрации, либо достаточно текущего `expires_at + used_by + revoked_at`.

2. Добавить серверную проверку этапа перехода по инвайту.
   Задача:
   При необходимости ввести endpoint валидации инвайта перед показом формы регистрации.

3. Уточнить поведение просроченного инвайта в UI.
   Задача:
   Если пользователь открыл страницу с уже истекшим кодом, форма должна явно показывать причину отказа.

4. Добавить интеграционные сценарии.
   Задача:
   Проверить:
   - открытие валидного инвайта;
   - регистрацию по валидному инвайту;
   - регистрацию по просроченному инвайту;
   - регистрацию по отозванному инвайту;
   - повторное использование инвайта.

### Критерий готовности

- Поведение ссылки инвайта однозначно и воспроизводимо.
- Нет неоднозначности между "перешел по ссылке" и "зарегистрировался по ссылке".

- `shared/native-core/crypto/*`

- `server/db/schema.go`

- `apps/desktop/src/main/kotlin/crypto/*`
- `apps/desktop/src/main/kotlin/viewmodel/AppViewModel.kt`

### Подзадачи

1. Выбрать модель.
   Задача:
   Зафиксировать одну из двух реализаций:
   - encrypted key backup с master password;
   - direct device-to-device secure transfer при link-device;
   - либо обе.

2. Спроектировать серверный контракт хранения backup.
   Задача:
   Сервер должен хранить только шифрованный backup blob, без знания мастер-ключа.

3. Реализовать экспорт ключевого материала в backup envelope.
   Задача:
   Включить:
   - identity keys;
   - signed prekeys;
   - sender keys;
   - ratchet state;
   - необходимые device metadata.

4. Реализовать восстановление на новом устройстве.
   Задача:
   После ввода master password или при secure link-device устройство должно получать ключевой материал и открывать историю.

5. Обновить UX.
   Задача:
   Добавить:
   - создание мастер-пароля;
   - индикатор наличия backup;
   - восстановление при привязке нового устройства.

6. Добавить тестовые векторы и интеграционные тесты.
   Задача:
   Проверить восстановление истории после перепривязки/добавления нового устройства.

### Подзадачи

1. Добавить security regression suite.
   Задача:
   Покрыть:
   - invite TTL;
   - remote wipe;
   - membership checks;
   - disabled external integrations in strict mode.

2. Добавить multi-device acceptance suite.
   Задача:
   Покрыть:
   - link-device;
   - key restore;
   - новый device с доступом к истории.

3. Добавить product acceptance suite по PRD.
   Задача:
   Свести основные user stories в один набор проверок.

### Критерий готовности

- Для каждого критичного замечания существует автопроверка или формализованный ручной smoke scenario.
